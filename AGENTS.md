# BlenderAnimtool working rules

- Ніколи не змінюй файли без прямого запиту користувача. Зберігай сторонні та незавершені зміни.
- Працюй з усіма коренями multi-root workspace, а не лише з поточною директорією.
- Не запускай subagents, background agents або делегування.
- Не вгадуй стан Blender чи поведінку DayZ. Перевіряй живий Blender; для engine-логіки спочатку звіряй `Research`, потім Ghidra, якщо доказів недостатньо.

## Live Blender

- Використовуй тільки `blender-remote` TCP. Не використовуй файлові bridge.
- Спочатку викликай `compact_query`; `execute_code` використовуй лише коли потрібної компактної операції немає.
- Завжди передавай `fields` або `item_fields`. Не запитуй matrices, constraints, custom properties чи повні списки без фактичної потреби.
- Спочатку фільтруй через `armature`, `bone`, `name_contains`, `data_path_contains`. Типовий `limit` — 20, `max_bytes` — 8192.
- Продовжуй через `result_page` лише якщо потрібна наступна сторінка.
- Для перевірки змін використовуй `snapshot_pose` перед операцією, `diff_pose` після неї та `restore_pose` для гарантованого відновлення замість повторного повного читання ріга.
- Контролюй фактичний розмір через `response_bytes`; якщо відповідь більша за 8192 байти, звужуй fields/filters до наступного запиту.
- Об'єднуй пов'язані короткі запити через `compact_batch` в один TCP round trip. Для вкладених даних використовуй поля на кшталт `constraints.name`. Не повторюй уже підтверджені дані.
- Для візуальної проблеми перевіряй screenshot; приховуй bone names і control shapes перед фінальним знімком, якщо вони заважають оцінці.

## Search and output limits

- Починай із точного `rg` з file glob і вузьким шаблоном. Не запускай рекурсивний широкий пошук без обмеження результатів.
- Не читай великий файл повністю, якщо достатньо `rg` і малого діапазону навколо збігу.
- Якщо відповідь інструмента може бути великою, спочатку отримай count/summary, потім потрібну сторінку або діапазон.
- У commentary повідомляй тільки новий факт, дію або blocker. У final не повторюй повну історію роботи.
- Після виконання повідомляй токени: точне значення, якщо лічильник доступний; інакше явно позначену приблизну оцінку.

## Changes and verification

- Роби мінімальні локальні patches; не запускай formatter на legacy-файлі цілком через малу зміну.
- Не встановлюй Python-пакети глобально. Використовуй `P:\BlenderAnimtool\.venv`; якщо його немає, один раз запусти `tools\bootstrap_dev.ps1`.
- Для перевірки змін запускай `tools\check_changed.ps1`; для встановлення та hot reload — `tools\install_blender_addons.ps1`.
- Спочатку запускай вузькі тести зміненого модуля. Повний suite запускай лише коли зміна має широкий вплив.
- Не стверджуй, що Blender-зміна працює, доки не перевірено встановлену копію і живу TCP-відповідь.
