## Meta ResourceGUID

confirmed from Workbench Ghidra evidence in
`../ghidra-raw/ghidra-raw-metafile-guid-pass-http.txt`:

- `FUN_1404305b0` registers `MetaFileClass.Name` as `Name&GUID` with
  `resourceNamePicker`.
- `FUN_140de0110` edits/displays this property as a resource name plus GUID and
  formats UI text with `"%s (GUID=%s)"`.
- `FUN_140454eb0` formats the stored 64-bit GUID value as exactly 16 uppercase
  hex characters.
- `FUN_140f96970` handles the duplicate-GUID diagnostic for two files sharing
  the same GUID.

Conclusion:

- `FE542D5525E19BD8` is an Enfusion `ResourceGUID` stored in the `.anm.meta`
  `Name` field as `{GUID}filename`.
- It is not an ANM binary chunk, not a checksum, and not derived from the
  validated ANM bytes in this pass.
- It is needed by Workbench's resource database to identify a resource
  independently from the path/name and to detect duplicate resource identities.
- `FUN_1404541e0` generates the GUID from the normalized resource name when no
  explicit `{GUID}` prefix exists. Reproducing that algorithm gives:

```text
StalkerSnork_runright.anm => FE542D5525E19BD8
stalkersnork_runright.anm => FE542D5525E19BD8
anm/fbx/StalkerSnork_runright.anm => 41868E2598EE0FBF
```

So this meta file's GUID is generated from resource name
`StalkerSnork_runright.anm`, not from full path, FBX contents, TXA contents, or
ANM bytes.

- empty/default frame blocks appear, e.g. `Scene_Root` has `$frame 0 24 { }`.

## EntityPosition Case

observed:

- The file contains `$node "entityposition"` and `$node "entityposition_end"`.
- It does not contain exact-case `$node "EntityPosition"`.

Ghidra evidence:

- `FUN_140e75c10` searches for exact string `EntityPosition` through
  `FUN_140e7bf50`.
- `FUN_140e7bf50` uses a direct name compare helper, not a documented
  case-insensitive compare in the current decompile.

Conclusion:

- likely: this sample's lowercase `entityposition` is treated as a normal node,
  not the special Workbench `EntityPosition` root-motion branch.
- Blender integration should export the exact-case `EntityPosition` node only
  when the special Workbench behavior is desired.

## Local TXA -> ANM Prototype Check

Commands run:

```powershell

dotnet run --project tools\TxaAnmPrototype -- txa2anm anm\fbx\StalkerSnork_runright.txa anm\fbx\StalkerSnork_runright.exact.from_txa.anm --mode exact
```

Conclusion:

- confirmed: for this FBX-derived Workbench TXA/ANM pair, local optimized
  TXA->ANM output is byte-identical to Workbench output.
- confirmed: this Workbench-produced ANM follows the optimized/TXA reduction
  path, not the exact/zero-threshold direct-FBX writer mode.
- confirmed: `.anm.meta` `ResourceGUID` does not affect ANM payload bytes.
